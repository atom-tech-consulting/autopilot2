# Fence `.cc-autopilot/tasks/` and `.cc-autopilot/insights/_index.md` from task-agent writes

## Goal

This task is anchored in goal.md's `## Done when` criterion: "An operator can point ap2 at a fresh project, paste a goal.md (with Mission + `## Done when`), and walk away for a week without intervention." The walk-away promise rests on daemon-owned state staying coherent across many task cycles; if a task agent can silently mutate the very audit/state files that gate verification and ideation, the walk-away clock can't be trusted — by the time the operator returns, the briefings the verifier read against may not match the briefings the operator authored. Two paths under `.cc-autopilot/` are daemon-owned audit/state surfaces but are NOT yet listed in `TASK_AGENT_FENCED_PATHS` (`ap2/tools.py:3638-3671`):

1. **`.cc-autopilot/tasks/`** — per-task briefing markdown files authored by the operator (via `ap2 add`) or by ideation (via `do_board_edit`'s add-* branch). The per-task verifier reads `## Verification` from these files at verification time. A task agent could (intentionally or by accident) rewrite its own briefing's Verification section mid-run to weaken the criteria the verifier evaluates against, OR edit an unrelated task's briefing and silently corrupt a future verification gate.

2. **`.cc-autopilot/insights/_index.md`** — auto-regenerated by `insights.maybe_regenerate_index(cfg)` (called pre-fire from `_maybe_ideate`, `ap2/insights.py`). Daemon-owned aggregated index of project-output insights. A task agent rewriting this file would corrupt the regeneration's input/output invariants and confuse ideation's Step 0.5 read.

The companion files in `insights/` — `<topic>.md` insight files themselves — STAY writable: `#evaluation`-tagged task agents legitimately CREATE / EDIT these per the ideation prompt's Step 0.5 contract ("write the result as a new file in `.cc-autopilot/insights/<topic>.md`"). Only `_index.md` is daemon-owned.

This advances goal.md's "No external mutation by task agents" constraint by closing two daemon-owned-but-unfenced gaps that match the exact failure mode TB-188 just guarded against for `ideation_proposals/`.

Why now: TB-188 landed the whole-directory fence for `ideation_proposals/` with explicit reasoning — "a malicious or confused agent could otherwise rewrite its own proposal's focus_anchor / why_now mid-run to cover scope drift." The same risk applies word-for-word to `tasks/` (agent could rewrite its own `## Verification` mid-run to weaken the criteria before the verifier reads them) and the same pattern applies to `_index.md` (daemon-regenerated state). Filing now means the fence-the-daemon-owned-state invariant lands while the TB-188 precedent is fresh; deferring lets the gap drift open as more daemon-owned files accumulate under `.cc-autopilot/`.

## Scope

- `ap2/tools.py` — add two entries to `TASK_AGENT_FENCED_PATHS` (line 3638-3671): `.cc-autopilot/tasks` (whole-directory, same style as `.cc-autopilot/ideation_proposals` already present at line 3670) and `.cc-autopilot/insights/_index.md` (single file path).
- `ap2/prompts.py::_TASK_HEADER` — update the "What the daemon and operator handle (do NOT touch)" list (lines 263-287) to mirror: add prose bullets for `.cc-autopilot/tasks/` (cite the same risk wording — "a task agent rewriting its own or another task's briefing could weaken verification mid-run") and for `.cc-autopilot/insights/_index.md` (cite the regeneration relationship — "daemon-regenerated index, individual `<topic>.md` insights remain writable"). The prompt header and `TASK_AGENT_FENCED_PATHS` must stay in sync (TB-110 invariant).
- `ap2/tests/test_tools.py` (or wherever existing fenced-paths tests live — locate via `grep -rn "TASK_AGENT_FENCED_PATHS" ap2/tests/`) — extend the existing test that walks `TASK_AGENT_FENCED_PATHS` and asserts each path produces an `Edit(<path>)` AND `Write(<path>)` entry in the SDK `disallowed_tools` list. New entries: assert the two additions show up.
- `ap2/tests/test_prompts.py` — extend the existing test that pins the prompt header's enumerated "do NOT touch" list. Assert both new paths appear by string match.
- `ap2/tests/test_tools.py` (or test_daemon if more appropriate) — new tests for the directory-fence semantics: an `Edit("/.cc-autopilot/tasks/foo.md")` attempt and an `Edit("/.cc-autopilot/insights/_index.md")` attempt are BOTH refused via `disallowed_tools` (mirror the TB-188 test for `ideation_proposals/`). Also pin that `Edit("/.cc-autopilot/insights/some_topic.md")` is NOT refused (the per-topic writes stay open).

## Design

### Why fence `tasks/` as a whole directory, not file-by-file

Same rationale TB-188 used for `ideation_proposals/`: briefings have content-dependent slug filenames (`<slug>.md`), so enumerating every file by name is impossible. The fence applies to the directory as a unit; the SDK's `Edit(<dir>)` / `Write(<dir>)` pattern propagates to any file under it. Pattern verified working for `ideation_proposals/` post-TB-188.

### Why fence `insights/_index.md` as a single file, not the whole insights/ directory

`#evaluation`-tagged task agents are EXPECTED to write new `<topic>.md` files into `insights/` per the ideation Step 0.5 contract. Fencing the whole directory would break that path. Only `_index.md` is daemon-owned (regenerated lazily by `insights.maybe_regenerate_index(cfg)`). The single-file fence is correct here even though the directory-fence is correct for `tasks/`.

### Prompt-header / disallowed_tools invariant

The two fence layers (Python-side `disallowed_tools` derived from `TASK_AGENT_FENCED_PATHS` + prompt-header prose enumeration) must stay in sync. TB-110's rationale is that an agent reading "this file is fenced" in the prompt should be told the truth — if disallowed_tools rejects an Edit but the prompt header doesn't list the file, the agent looks confused at runtime. Both layers updated in lockstep.

### Backwards compatibility

Additive only — no removal of existing fence entries, no behavior change for paths not in the new entries. Operators who manually edit briefings via the filesystem are unaffected (they're not task agents; the SDK fence applies only inside `sdk.query` calls).

### Existing legitimate operator-flow check

`ap2 add` writes a new briefing file in `tasks/` via `do_board_edit({"action": "add_backlog", ...})`. That writes happens in the daemon's drain handler / CLI codepath, NOT inside a task agent's `sdk.query` call. The fence applies only to task-agent SDK calls; daemon/CLI/MCP-tool code paths bypass the SDK guard entirely. No legitimate flow breaks.

## Verification

- `uv run pytest -q ap2/tests/` — full regression gate passes.
- `python3 -c "from ap2.tools import TASK_AGENT_FENCED_PATHS; assert '.cc-autopilot/tasks' in TASK_AGENT_FENCED_PATHS; assert '.cc-autopilot/insights/_index.md' in TASK_AGENT_FENCED_PATHS"` — both entries land in the fenced-paths tuple.
- `grep -qE 'cc-autopilot/tasks' ap2/prompts.py` — prompt header's enumerated list mentions tasks/.
- `grep -qE 'insights/_index\.md' ap2/prompts.py` — prompt header mentions _index.md.
- prose: a test in `test_tools.py` (or wherever the fenced-paths-tooling-derivation test lives — locate via `grep -rnE "TASK_AGENT_FENCED_PATHS" ap2/tests/`) asserts the SDK `disallowed_tools` list generated from `TASK_AGENT_FENCED_PATHS` contains BOTH `Edit(.cc-autopilot/tasks)` AND `Write(.cc-autopilot/tasks)`, AND `Edit(.cc-autopilot/insights/_index.md)` AND `Write(.cc-autopilot/insights/_index.md)`.
- prose: a test pins the prompt header's "do NOT touch" list contains a bullet mentioning `.cc-autopilot/tasks` (or `tasks/`) AND a bullet mentioning `insights/_index.md`. Existing entries (TASKS.md, CLAUDE.md, operator_queue.jsonl, etc.) remain enumerated — pin the count of bullets in the section to detect accidental removals.
- prose: a test pins the directory-fence semantics for `tasks/` — given a path string like `.cc-autopilot/tasks/some-slug.md`, assert the `disallowed_tools` rejection logic / fence-prefix-match logic treats it as fenced (mirror the existing test for `ideation_proposals/<TB-N>.json` from TB-188).
- prose: a test pins that the per-topic `<topic>.md` writes stay OPEN — given a path like `.cc-autopilot/insights/sharpe_floor.md`, the fence logic does NOT reject it (this is the legitimate `#evaluation`-task-agent write target per the ideation prompt's Step 0.5 contract).

## Out of scope

- Fencing the other small daemon-owned files under `.cc-autopilot/` (`cron_state.json`, `mm_state.json`, `retry_state.json`, `auto_diagnose_state.json`, `context.json`, `env`, `daemon.{log,pid}`, `.gitignore`). Lower-risk; defer to a separate sweep TB or a follow-up if friction observed. Same architectural pattern — additive entries to `TASK_AGENT_FENCED_PATHS` plus the prompt-header mirror.
- Fencing `.cc-autopilot/debug/` (per-run dump files post-TB-165/166). Different ownership model — agent reads its own run dumps don't yet exist at task-run start; fencing might create a chicken-and-egg with how dumps are written. Separate TB if pursued.
- Restructuring `TASK_AGENT_FENCED_PATHS` from a flat tuple into a more typed structure (e.g., `dataclasses.dataclass` with fence kind + path). Flat tuple is fine for now; refactor when the count crosses ~20 entries or a new fence kind is needed.
- Adding a CLI lint that warns the operator if they hand-edit a briefing while a task on the same TB is in flight. Out of scope — operator-side workflows are unconstrained by the SDK fence.
- Migrating existing on-disk briefings or insight-index files. The fence is forward-looking only.
- Adding a TB-110-style post-hoc snapshot rollback check for `tasks/` and `_index.md`. The existing `rollback._VIOLATION_CHECK_EXCLUDED_PATHS` mechanism would catch them automatically via the new `TASK_AGENT_FENCED_PATHS` entries; no additional rollback wiring needed.

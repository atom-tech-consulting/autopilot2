## Goal

Close the final four of the twelve CLI-verb coverage-debt names that TB-209's `ap2/tests/test_coverage_drift.py` docstring (L401-413) explicitly enumerates as a follow-up to TB-209's drift-gate landing. The four sandbox audit/setup verbs — `ap2 sandbox project-audit`, `ap2 sandbox project-setup`, `ap2 sandbox user-audit`, `ap2 sandbox user-setup` — are registered in `ap2/cli.py`'s `build_parser()` sandbox subcommand group (verified via `_collect_cli_verbs` in `ap2/tests/_source_registry.py` extracted by TB-209) but the only substring reference under `ap2/tests/` is the comment-block shim itself (`Grep ap2/tests/` for any of the four returns one file: `test_coverage_drift.py`). The substring drift gate passes by gate-satisfaction shim, not by any real assertion of CLI behavior. Direct mirror of TB-205's approved pattern (4 SDK-cost env knobs → focused per-name contract tests) and TB-210/TB-213's closure shape (4 names + comment-block shim removal), rotated onto the sandbox audit/setup subset of the CLI-verb axis. Goal anchor: **Current focus: code quality** (goal.md L38) — Testing coverage axis (goal.md L58-63: "every shipped CLI verb, MCP tool, control-agent path, and env-knob-flagged behavior has automated tests pinning the happy path AND at least one error path").

Why now: TB-209's docstring names this gap as a closed-set follow-up at exactly 12 entries that bifurcate cleanly along verb-family — 4 daemon-lifecycle (TB-213, in flight) + 4 sandbox install-* (TB-214, sibling) + 4 sandbox audit/setup (this TB). Together with the two siblings, this TB closes the 12-verb debt in one drop. The audit/setup verbs are the pre-onboarding readiness-check surface (project-audit / user-audit return WARN/FAIL diagnostics before a daemon ever starts; project-setup / user-setup are the corresponding fix paths). Without real assertions, every future refactor of `cli.cmd_sandbox_project_audit` / `cmd_sandbox_user_audit` and the underlying `sandbox.project_audit` / `sandbox.user_audit` helpers can silently break the readiness-check contract (e.g. a check that should FAIL stops FAILing, or a fix path stops fixing) while the drift gate stays green via the shim. Replacing the shim with real tests closes the sandbox audit/setup subset of the testing-axis gap that TB-209 explicitly tagged as "coverage debt, not exemptions" — finalizing TB-209's enumerated closure.

## Design

- Mirror `ap2/tests/test_sandbox.py`'s existing pattern for sandbox audit/setup tests: `test_sandbox.py` already has tests for related sandbox surfaces (TB-124 added `_user_login_shell` tests; TB-125 added project_audit git-identity FAIL tests). Implementer reads those for the established stubbing approach (fake home dirs, subprocess mocks, `Path` patching).
- For `ap2 sandbox project-audit`: handler in `ap2/cli.py` dispatches to `sandbox.project_audit` (`Grep "project_audit\|project-audit" ap2/cli.py ap2/sandbox.py` to locate the exact symbol). Happy path: all readiness checks PASS on a stubbed-clean project; error path: at least one documented FAIL branch fires (e.g. missing git identity per TB-125, or missing daemon-fenced dir, or missing OAuth token in target user's shell).
- For `ap2 sandbox project-setup`: writes the per-project sandbox state (initial CLAUDE.md, git config, fenced dirs). Happy path stubs the writes and asserts each side-effect; error path covers existing project / unwritable target / git clone failure.
- For `ap2 sandbox user-audit`: handler dispatches to `sandbox.user_audit`. Happy path: all per-user readiness checks PASS; error path: at least one documented FAIL or WARN branch fires (e.g. missing OAuth token, ap2 not on $PATH).
- For `ap2 sandbox user-setup`: creates the sandbox user, sets shell, writes initial config. Happy path stubs the user-creation subprocess and asserts the expected useradd / config-write side-effects; error path covers duplicate user / unwritable home / useradd subprocess failure.
- Test-function naming convention: `def test_cmd_sandbox_<verb_underscored>_<aspect>(...)` — e.g. `test_cmd_sandbox_project_audit_happy_path`, `test_cmd_sandbox_user_setup_duplicate_user`. This convention is what the auto-verify bullets grep for.
- The substring drift gate's CLI-verb walk reads `_collect_cli_verbs()` from `ap2/tests/_source_registry.py` (extracted in TB-209), so the test file's reference to the verb name (e.g. `"ap2 sandbox project-audit"` as a string literal OR `cmd_sandbox_project_audit` as a symbol) suffices for the gate's `name in blob` check. Pin the four full verb strings (`"ap2 sandbox project-audit"`, etc.) somewhere — docstring, test data, or commented justification — for the most robust reference.
- Removing the 4 comment-block rows (L408-411) is a 4-line deletion in `test_coverage_drift.py`. The drift gate continues to pass because the new test module references the verb names.
